# Intellivix Data Studio — 유지보수 가이드

인수인계 받은 유지보수 담당자를 위한 기술 문서입니다.

---

## 1. 프로젝트 개요

- **역할**: 이미지 라벨링(YOLO 바운딩 박스) 및 VLM 검수(수용/거절) 작업을 관리·배정·검수하는 웹 도구.
- **역할 구분**
  - **관리자(ADMIN)**: 프로젝트 생성, 데이터 동기화, 작업자/폴더 배정, VLM JSON import·배분, 검수.
  - **작업자(WORKER)**: 대시보드에서 배정된 폴더 선택 후 YOLO 또는 VLM 작업 수행 및 제출.
  - **검수자(REVIEWER)**: 관리자 계정에서 Worker View / Reviewer View 전환 시, 제출된 VLM 작업 검수(수용/반려).

---

## 2. 개발 환경

- **Node.js**: 18+ 권장.
- **의존성 설치**: `npm install`
- **개발 서버**: `npm run dev` (Vite dev server, API는 동일 서버의 미들웨어로 처리)
- **빌드**: `npm run build` → `dist/` 산출
- **환경 변수**: 필요 시 `.env.local` 등에 설정 (예: `GEMINI_API_KEY` — VLM 관련 기능에서 사용할 수 있음)

## 4. 데이터베이스

- **파일**: 프로젝트 루트의 `datasets.db` (SQLite).
- **저널 모드**: WAL (`db.pragma('journal_mode = WAL')`).
- **주요 테이블**
  - `tasks`: YOLO용 작업 (id, folder, imageUrl, assignedWorker, status, sourceType, sourceData 등).
  - `vlm_tasks`: VLM 검수용 작업 (동일 스키마, sourceType='vlm-review').
  - `folder_metadata`: 폴더별 assignedWorker, memo 등.
  - `logs` / `vlm_logs`: 작업별 액션 로그 (START, SAVE, SUBMIT, APPROVE 등).
  - `task_issues`: 이미지 이슈 요청/해결.
  - `deleted_tasks`: 삭제된 작업 ID·타임스탬프 (동기화용).
- **스키마 변경**: `vite.config.ts` 상단 `db.exec(...)` 및 마이그레이션 블록에서 처리. UNIQUE 제거 등은 기존 마이그레이션 참고.

---

## 5. 주요 API (vite.config.ts 내부)

- `POST /api/login`: 로그인 (users.json 기반).
- `GET /api/projects`, `POST /api/projects`, `DELETE /api/projects`: 프로젝트 목록·생성·삭제.
- `GET /api/projects/detail`: 프로젝트 상세 (폴더별 통계, VLM 시 vlm_tasks 집계).
- `GET /api/projects/overview`: 대시보드용 프로젝트·폴더 요약.
- `GET /api/datasets`: 작업 목록 (페이징, worker/folder 필터, `ids` 쿼리로 ID 목록 조회).
- `GET /api/sync/delta`: 특정 시점 이후 변경·삭제된 작업 (동기화용).
- `POST /api/sync`: 파일 스캔 → DB 반영. **`projectId` 또는 `folders` 쿼리로 범위 지정** (권장). 전체 `datasets` 스캔은 **`?full=1`** (느림).
- `POST /api/assign-worker`: 폴더/작업자 배정 (YOLO·VLM 폴더 공통).
- `POST /api/plugins/vlm/import-json`: VLM JSON import.
- `GET /api/plugins/vlm/import-json/files`: import 가능한 JSON 파일 목록.
- `POST /api/plugins/vlm/assign`, `POST /api/plugins/vlm/unassign`: VLM 작업 건수 단위 배분/해제.
- `POST /api/task-commit`, `POST /api/save`: 작업 상태·라벨·VLM sourceData 저장.
- `GET /api/logs`, `GET /api/analytics/*`: 로그·집계.

## 6. 설정·데이터 경로

- **프로젝트 정의**: `_projects.json` (vite.config.ts에서 경로 resolve).
- **이미지/라벨**: `datasets/` 하위 **프로젝트 폴더** 중심 권장. 동기화 시 루트의 작업자명 폴더로는 더 이상 자동 배정하지 않음; 작업자는 매핑/수동 배정.
- **가이드 PDF**: `public/guides/` + `public/guides/list.json` (제목·파일명 목록). 상단 Guide 메뉴에서 로드.

---

## 7. 배포

- `npm run build` 후 `dist/`를 정적 서버에 호스팅하면 **프론트만** 제공됨.
- **API·DB·Sync**는 현재 **Vite dev 서버 미들웨어**에만 구현되어 있으므로, 프로덕션에서는 동일 로직을 Node/Express 등 별도 서버로 이전하거나, Vite preview 시 `vite.config.ts`의 미들웨어가 실행되도록 구성해야 함.

---

## 8. 트러블슈팅

- **DB 잠금/느린 Sync**: WAL 모드 사용 중. 동시 쓰기 많을 경우 `wal_checkpoint` 등 확인. Sync는 대량 파일 스캔 시 부하 가능.
- **VLM 배분 후 목록에 안 보임**: 클라이언트 캐시(cachedTasks) 미반영. 배분 API가 반환하는 `taskIds`로 `/api/datasets?ids=...` 호출 후 병합하는지 확인 (storage.ts `assignVlmTasks`).
- **프로젝트 삭제해도 VLM 작업이 배정된 채로 남음**: `DELETE /api/projects`에서 해당 프로젝트의 `vlmSourceFile`에 연결된 `vlm_tasks`의 assignedWorker·status 초기화 여부 확인.
- **검수 시 제출 건이 안 보임**: 관리자 쪽 캐시에 해당 작업자 작업이 없을 수 있음. 검수 버튼에서 `fetchAndMergeWorkerTasks(workerName)` 호출 후 첫 SUBMITTED 작업 열기 로직 확인 (Dashboard.tsx).
